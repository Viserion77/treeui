<Title>Roadmap</Title>

TreeUI ships components in milestones, grouped by dependency depth and product impact.
Each milestone builds on the previous one while keeping the public API compact and composable.
Not every repeated UI shape becomes a public export. When a pattern is mostly layout or styling,
TreeUI prefers documenting a composition recipe on top of existing primitives.

---

## Admission filter

A repeated app pattern graduates into a public component when it:

- adds reusable interaction, accessibility, or state behavior
- carries domain-agnostic semantics that can travel across products
- removes real API friction beyond what `TCard`, `TBadge`, `TButton`, and layout primitives already cover

## Current state — Public surface (shipped)

Milestone 1 and Milestone 2 now ship as part of the public TreeUI surface.
The next round of work moves from closing core component gaps to documenting composition recipes,
hardening conventions, and choosing the next future-facing milestone.

| Form | `TInput`, `TTextarea`, `TCheckbox`, `TCombobox`, `TFileUpload`, `TRadio`, `TRadioGroup`, `TSelect`, `TMultiSelect`, `TNumberInput`, `TDatePicker`, `TSwitch`, `TFormField`, `TToggleGroup`, `TMarkdownEditor` |

## Table direction

`TTable` stays intentionally lean and focused on structured listing. Pagination, filters, toolbars, bulk actions, and data fetching stay composable outside the table, with `TPagination` used separately when needed.

## Milestone 1 — Application Primitives

Status: shipped.

Close the highest-friction gaps for SaaS products, dashboards, admin panels, and internal tools.

| Component | Description |
|---|---|
| `TStat` | Compact metric block with label, value, and optional trend or meta content for dashboards and summaries |
| `TMultiSelect` | Multiple selection for filters, tags, teams, permissions, and bulk actions |
| `TNumberInput` | Numeric field with increment and decrement stepper controls, min/max, and step behavior |
| `TSelectableList` | Persistent single-select list with active state, keyboard navigation, and flexible item slots for templates, events, guides, and goals |
| `TTreeView` | Hierarchical navigation and nested selection for files, categories, and permissions |
| `TSteps` / `TStepper` | Step-by-step progress pattern for onboarding, wizards, and setup flows |
| `TToggleGroup` | Compact single and multi-select toggle buttons as a visual alternative to radios and checkboxes |

## Milestone 2 — App Shell & Layout

Status: shipped.

Add layout helpers and application shell building blocks without forcing a full page framework.

| Component | Description |
|---|---|
| `TSidebar` / `TNavMenu` | Collapsible lateral navigation for dashboards and workspace layouts |
| `TNavbar` / `TAppBar` | Top-level application navigation and global action bar |
| `TContainer` | Width-constrained layout wrapper for content sections |
| `TGrid` | Semantic grid helper for dashboard cards, stat groups, forms, and responsive layouts |
| `TStack` | Vertical and horizontal spacing primitive for common flex layouts, section headers, and inline actions |

## Pattern track — Docs-first compositions

Some repeated app UI should land as documented recipes instead of standalone exports.
These patterns stay important, but TreeUI keeps them composable until they prove they need
their own public API. The recipes now live in docs as guidance rather than new package surface.

| Suggested pattern | TreeUI direction | Delivery status |
|---|---|---|
| Stat group (`TreeStatGroup`) | Document as `TStat` + `TGrid`; no standalone `TStatGroup` planned | Delivered as docs recipe |
| Section header (`TreeSectionHeader`) | Document as heading + `TStack` + optional `TBadge` or `TButton`; no dedicated export planned | Delivered as docs recipe |
| Subpanel (`TreePanel`) | Prefer `TCard` composition, especially `variant="soft"`; revisit only if a distinct semantic API emerges | Delivered as docs recipe |
| Stacked card (`TreeStackCard`) | Use `TCard` slots plus `TTag`, `TBadge`, and actions; revisit later only if a stable slot contract repeats across products | Delivered as docs recipe |
| Eyebrow text (`TreeEyebrow`) | Keep as a typography recipe for now; revisit when TreeUI has a broader text primitive strategy | Delivered as docs recipe |
| Form stack (`TreeFormStack`) | Covered by `TStack`, `TGrid`, and `TFormField`; no extra layout wrapper planned | Delivered as docs recipe |
| Ranking (`TreeRanking`) | Defer as a narrower data-display recipe until the dashboard/data pack grows | Delivered as docs recipe |
| Action panel (`TreeActionPanel`) | Keep app-level for now; usually composes from `TAlert`, `TCard`, and `TButton` | Delivered as docs recipe |

## Candidate review — Recent suggestions

The recent component suggestions now map to two resolved buckets:

- Promote portable primitives: `TStat` and `TSelectableList` shipped
- Keep higher-level page assemblies as documented recipes in `Foundation/Patterns` and `Patterns/Recipes`

## Aliases & conventions

TreeUI keeps naming intentionally predictable across docs and exports:

- Primary public names use the `T` prefix.
- `Tree<Name>` exports remain available for compatibility and migration.
- `TNavbar` / `TAppBar` and `TSteps` / `TStepper` are export aliases for the same implementation.
- Common product terms like `Snackbar` stay documentation aliases only and do not create duplicate exports.

Some common UI terms are intentionally documented as aliases to existing patterns rather than planned as separate components.

| Common term | TreeUI pattern | Convention |
|---|---|---|
| `Snackbar` / `Notification` | Use `TToast` | Semantic alias only |
| `Banner` | Use `TAlert` | Semantic alias only |
| `Collapsible` / `Details` | Use `TAccordion` in single mode with `collapsible` enabled | Semantic alias only |
| `App bar` | Use `TNavbar` or `TAppBar` | Export alias pair |
| `Stepper` | Use `TSteps` or `TStepper` | Export alias pair |

## Future

- `TCarousel` for focused media and card browsing patterns
- Data visualization primitives for dashboards, including charts and sparklines
- Additional framework packages (`@treeui/react`, `@treeui/svelte`) reusing the same token system, naming, accessibility guidance, and design language.
- Motion/animation token expansion.
- Right-to-left (RTL) layout support.
- Theme builder / visual token editor.
